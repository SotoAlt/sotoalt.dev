lepong: a world model plays pong in the browser

2026.04 · 8 min

tl;dr

A 13M-parameter CNN JEPA with a frozen backbone and a 1,930-parameter Linear(192, 10) state head runs right here in your browser via ONNX Runtime Web -- ~50 MB model, no server, no GPU. Trained on 30K Pong frames, it decodes ball position at 2.45% median error and plays interactively at ~6 Hz inference. Under occlusion (toggle with q/w/e/r above) the baseline model collapses -- we fixed it with data augmentation. A pretrained DINOv2 with zero Pong training handles occlusion 4--7x better at the embedding level, but loses at the task because it has no predictor. Robustness is not task precision.

what this demo is

The model running above is a real neural network. 4-layer CNN encoder, 6-layer Transformer predictor, Linear state head. It takes 3 frames of 128x128 pixel history, predicts the next latent state, and reads off ball_y to move the left paddle toward the predicted intercept. No simulator, no physics cheats -- just pixels in, paddle target out.

The entire 13M-parameter model was exported to ONNX and runs client-side in your browser via WebAssembly. On a laptop this takes 15--40 ms per inference at ~6 Hz. The game physics run at ~60 fps on the main thread; every 5th tick, the current frame is captured onto a 128x128 offscreen canvas that exactly replicates PongWorld.render() from the Python training code, then fed through the network.

The left paddle (cyan label, "JEPA") is controlled entirely by the network's predicted ball_y. The right paddle (amber, "YOU") is your mouse. A faint cyan ghost ball shows where the JEPA thinks the ball is right now. When the ball approaches your side and the JEPA has a prediction, a glowing target zone appears -- match your paddle to the zone for a precision rating.

the obliteration

We built a v1 in early April 2026 and shipped it as a "world model demo". Then we re-read our own code and found twenty flaws. The server re-rendered from state the client already sent -- a round-trip decoder pretending to be a world model. The encoder was labeled "frozen" in every document but requires_grad_(False) was never called in the training script. The "CEM planner" was literally argmin(abs(candidate - paddle_y)). The A/B test was rigged: both paddles ran the same trivial tracker.

We wrote an obliteration post-mortem in the research journal, burned the v1 labels, and rebuilt v2 from scratch: client renders pixels, server only sees PNG bytes, encoder genuinely frozen with asserted requires_grad checks, classical extrapolator computed alongside for honest comparison. The demo above is the v2 architecture, now running entirely client-side.

three OOD axes

We evaluated out-of-distribution robustness along three axes, adapting the stable-worldmodel protocol from Galilai Group (the standard used by DINO-WM, la-leWM, and LeWM papers).

OOD axis method result
renderer shift byte-compare client JS vs Python render 99.9% identical (no shift)
state trajectory SWM controller on novel trajectories 99.3% in-dist, 88.7% OOD
pixel occlusion right-side blackout 0%/20%/40%/60% state head collapses at 40%+

The renderer shift axis is trivially passed because the JS canvas render is a line-by-line port of the Python PongWorld.render(). State trajectory shift shows a real but modest drop: the controller trained on AI-vs-AI rallies degrades when facing human play patterns. Pixel occlusion is where it gets interesting.

the augmentation fix

Pixel occlusion at 40% meant the state head saw embeddings it had never encountered during training, and its predictions went haywire. The fix was simple: regenerate the training data with random right-side blackout applied during collection. 30K frames with occlusion drawn uniformly from 0--60%. Retrain only the 1,930-parameter state head (backbone stays frozen).

metric before augmentation after augmentation change
ball_x error @ 40% occ high (unusable) moderate -58%
SWM controller success 46.2% 62.5% +35%
ball_y OOD drop +125% +6% nearly eliminated

Surprise result: occlusion augmentation also fixed state-trajectory OOD. The ball_y decoder drop on novel trajectories went from +125% to +6%. Apparently, training the state head on partially-observed embeddings taught it a more robust decoding of the latent space, even for fully-visible inputs with unseen dynamics.

the DINOv2 foil

A natural question: does a pretrained foundation model handle occlusion better than our small task-specific CNN? We tested DINOv2 ViT-S/14 (21.6M parameters, pretrained on LVD-142M images, zero Pong training) as a foil.

model cos sim @ 0% occ cos sim @ 40% occ drop
DINOv2 ViT-S/14 1.00 0.94 -6%
our CNN JEPA 1.00 0.20 -80%

DINOv2 barely flinches at occlusion -- its embeddings at 40% blackout are still 0.94 cosine to the clean version. Our CNN's embeddings collapse to 0.20. But in live gameplay our CNN wins, because it has a predictor. The JEPA architecture doesn't just encode the current frame: it predicts where the ball will be at the next timestep. DINOv2 has no temporal model, so even if its embeddings survive occlusion, it can only decode "where the ball is now" (if visible) rather than "where it's going". The CNN aims at the future.

The lesson: robustness lives in pretrained features, but task precision needs the prediction architecture. The ideal system would be DINOv2 encoder + JEPA predictor + state head -- this is exactly what Facebook's jepa-wms architecture provides, and it's what we plan to test next.

reproduce

The full codebase is at github.com/SotoAlt/aura. Key files: 

# Training data (30K frames, AI vs AI with noise)
python -m world_model.envs.pong_world --episodes 300 --steps 100

# Train CNN JEPA + state head (RunPod L4, ~12 min, ~$0.20)
python scripts/train_pong_state_head_frozen.py \
    --data data/pong_v1.npz --epochs 50

# Export to ONNX
python scripts/export_onnx.py \
    --checkpoint checkpoints/jepa_pong_statehead_frozen.pt

# This page loads the ONNX from ../assets/lepong/pong_jepa.onnx
# and runs inference entirely in the browser.

The ONNX model is ~50 MB (13M parameters at float32). Inference runs via WebAssembly in any modern browser -- no GPU, no server, no Python. The game physics, rendering, and neural network all execute on the client.
